# Product Reviews System — Design Spec

## Overview

Add a product reviews system to the e-commerce platform. Customers with verified purchases can leave 1-5 star ratings with text comments. Reviews require admin approval before becoming publicly visible. Approved reviews enhance SEO through schema.org structured data (AggregateRating + Review).

## Requirements Summary

- **Rating:** 1-5 stars (integer) + text comment (10-500 chars)
- **Authentication:** User must be logged in (Firebase Auth)
- **Verified purchase:** User must have an order with `paymentStatus === "paid"` and `fulfillmentStatus === "delivered"` containing the product
- **One review per product per user**
- **Admin moderation:** Reviews start as `pending`, admin approves or rejects silently
- **Rejection:** Silent — no notification to the user
- **Display name:** Anonymized/partial (e.g., "J. M.") — generated at render time from stored `displayName`
- **Aggregated stats:** Average rating + total review count shown on product page
- **SEO:** schema.org AggregateRating and Review in JSON-LD
- **Security:** Server-side sanitization of comment text, validation of all inputs
- **i18n:** UI translated to es, en, zh-HK. Review content stays in original language.
- **Responsive:** Consistent with existing shop styles via Tailwind CSS

## 1. Data Model

### Collection: `reviews`

| Field            | Type     | Description                                      |
|------------------|----------|--------------------------------------------------|
| id               | string   | Auto-generated by Firestore                      |
| productId        | string   | Reference to the product                         |
| userId           | string   | Firebase Auth UID                                |
| displayName      | string   | User's display name (for partial name generation)|
| rating           | number   | Integer 1-5                                      |
| comment          | string   | Sanitized plain text, 10-500 characters          |
| verifiedPurchase | boolean  | Always `true` (enforced at creation)             |
| status           | string   | `"pending"` | `"approved"` | `"rejected"`       |
| createdAt        | Date     | Timestamp of creation                            |
| updatedAt        | Date     | Timestamp of last status change                  |

### Product document — new fields

```typescript
reviewStats: {
  averageRating: number   // Rounded to 1 decimal
  totalReviews: number    // Count of approved reviews only
}
```

`reviewStats` is recalculated whenever a review is approved or rejected via the admin API.

### Partial name generation

Computed at render time (not stored):
- "Juan Martinez" → "J. M."
- "Ana" → "A."
- Null/empty → "Anon."

## 2. API Routes

### New infrastructure: `verifyUser()` helper

A new `verifyUser()` utility is needed in `/src/lib/user-auth.ts` (similar to the existing `verifyAdmin()` in `admin-auth.ts`). It verifies the Firebase Auth token from the `__session` cookie and returns the authenticated user's UID and profile. This is new infrastructure — no existing public API route currently authenticates regular users server-side.

### Public routes (require user auth via `verifyUser`)

#### `POST /api/reviews`

Create a new review.

**Request body:**
```json
{
  "productId": "string",
  "rating": 1-5,
  "comment": "string (10-500 chars)"
}
```

**Validation chain:**
1. Authenticate user via Firebase Auth token
2. Verify the user has an order with `paymentStatus === "paid"` and `fulfillmentStatus === "delivered"` containing the product (query `orders` collection)
3. Verify the user has not already reviewed this product (query `reviews` collection)
4. Validate rating: integer, 1 ≤ rating ≤ 5
5. Sanitize comment: strip HTML tags, trim whitespace, validate 10-500 chars
6. Create review document with status `"pending"`

**Response:** 201 with review ID

**Error responses:**
- 401: Not authenticated
- 403: No verified purchase / already reviewed
- 400: Invalid rating or comment

#### `GET /api/reviews?productId=xxx`

List approved reviews for a product.

**Query params:**
- `productId` (required)
- `page` (optional, default 1)
- `limit` (optional, default 10)

**Returns:** Only reviews with status `"approved"`, ordered by `createdAt` descending. Uses offset-based pagination consistent with existing API patterns.

### Admin routes (require `verifyAdmin`)

#### `GET /api/admin/reviews`

List reviews with filtering.

**Query params:**
- `status`: `"pending"` | `"approved"` | `"rejected"` | `"all"` (default: `"pending"`)
- `page` (optional, default 1)
- `limit` (optional, default 20)

**Returns:** Reviews with product name included for context.

#### `PUT /api/admin/reviews/[id]`

Approve or reject a review.

**Request body:**
```json
{
  "status": "approved" | "rejected"
}
```

**Side effect:** Recalculates and updates `reviewStats` on the product document inside a **Firestore transaction** to prevent race conditions:
1. Query all approved reviews for the product
2. Calculate average rating (rounded to 1 decimal) and total count
3. Update the product document's `reviewStats` field atomically

**Response:** 200 with updated review

## 3. SEO — Schema.org Structured Data

The existing `ProductJsonLd` component (`/src/components/product/product-jsonld.tsx`) is extended to accept `reviewStats` and `reviews` as props. The component will include:

```json
{
  "@type": "Product",
  "...existing fields",
  "aggregateRating": {
    "@type": "AggregateRating",
    "ratingValue": 4.3,
    "reviewCount": 12,
    "bestRating": 5,
    "worstRating": 1
  },
  "review": [
    {
      "@type": "Review",
      "author": { "@type": "Person", "name": "J. M." },
      "datePublished": "2026-03-15",
      "reviewRating": {
        "@type": "Rating",
        "ratingValue": 5
      },
      "reviewBody": "Excelente calidad..."
    }
  ]
}
```

- Only included when `reviewStats.totalReviews >= 1`
- `aggregateRating` sourced from precalculated `reviewStats` on the product document
- `review` array includes all approved reviews displayed on the page

### SEO Impact

- **Rich snippets:** Google displays star ratings in search results when AggregateRating markup is present
- **User-generated content:** Reviews add unique, relevant text to each product page, improving organic ranking
- **E-E-A-T signal:** "Verified purchase" reviews strengthen Experience, Expertise, Authoritativeness, Trustworthiness signals

## 4. Frontend — Product Page

### Aggregated summary

Positioned below the price, before the product description.

- Star icons (filled/empty based on average) + numeric rating (e.g., "4.3") + review count (e.g., "12 reseñas")
- Clicking scrolls to the reviews section
- If no reviews: "Sin reseñas todavía"

### Reviews section

Positioned below the product description.

- Heading: "Reseñas de clientes"
- Each review card shows:
  - Star rating (1-5 icons)
  - Partial name ("J. M.")
  - "Compra verificada" badge
  - Relative date (e.g., "hace 3 días")
  - Comment text
- Pagination: "Cargar más" button, 10 per batch
- Sorted by most recent first

### Review form

Conditional display:
- **Not logged in:** "Inicia sesión para dejar una reseña"
- **Logged in, no purchase:** "Solo los compradores verificados pueden dejar reseñas"
- **Logged in, already reviewed:** "Ya has dejado una reseña para este producto"
- **Eligible:** Show form

Form fields:
- Star selector: interactive 1-5 stars (click/tap to select)
- Textarea: max 500 characters with live counter, min 10 characters
- Submit button: "Enviar reseña"

Post-submit: Toast notification — "Tu reseña ha sido enviada y será visible tras su aprobación"

Client-side validation: rating required, comment 10-500 chars.

### Responsive behavior

- Desktop and mobile: vertical list layout, full-width form
- Stars: minimum 44px touch target on mobile
- Textarea: 100% width on all breakpoints
- Consistent Tailwind styling: same colors, typography, spacing, rounded corners, shadows as the rest of the shop

## 5. Backoffice — Admin Reviews

### Navigation

New "Reviews" entry in the admin sidebar at `/zr-ops/reviews/` (English label, consistent with existing sidebar entries like "Products", "Orders", etc.).

### List view

Table with columns: Producto, Usuario, Rating (stars), Comentario (truncated), Fecha, Estado.

**Filters:** Tabs or dropdown for status — Pendientes (default) / Aprobadas / Rechazadas / Todas.

**Actions per row (contextual):**
- Pending: "Aprobar" + "Rechazar" buttons
- Approved: "Rechazar" button (to retract)
- Rejected: "Aprobar" button (to reconsider)

**Interactions:**
- `ConfirmDialog` before approve/reject (never browser confirm/alert)
- Toast feedback after each action
- Expandable row or click-to-modal for full comment text
- Pagination consistent with existing admin pages

**No separate detail page** — the data is simple enough (stars + text) to handle in the list view.

### Pattern

Follows the structure and conventions of existing admin pages (campaigns, templates, products): same layout, same component patterns, same styling.

## 6. Security

### Input sanitization (server-side)

- Strip all HTML tags via regex (no external library needed — content is plain text only)
- Trim leading/trailing whitespace
- Validate length: 10-500 characters
- Validate rating: integer, 1 ≤ rating ≤ 5

No `dangerouslySetInnerHTML` is ever used for review content. React JSX escapes text by default.

### Authentication & authorization

- User routes: verify Firebase Auth token from request
- Verify `userId` from token matches the submitting user (prevent spoofing)
- Verified purchase: query `orders` for productId + userId + `paymentStatus === "paid"` + `fulfillmentStatus === "delivered"`
- Uniqueness: query `reviews` for productId + userId to prevent duplicates
- Admin routes: `verifyAdmin()` middleware (existing pattern)
- Admin can only change status, not edit content

### Rate limiting

- Max 5 review submissions per hour per user (server-side check on `reviews` collection by userId + createdAt)
- Prevents abuse from users with multiple purchases across products

### Firestore composite indexes

Required indexes for the `reviews` collection:
- `productId` ASC, `status` ASC, `createdAt` DESC — for public listing of approved reviews per product
- `status` ASC, `createdAt` DESC — for admin listing filtered by status
- `productId` ASC, `userId` ASC — for duplicate review check

### Rendering

- Comments rendered as plain text in JSX (auto-escaped by React)
- No HTML rendering of user-generated content

## 7. Internationalization (i18n)

### New translation keys

**Namespace `reviews`** (in es.json, en.json, zh-HK.json):
- Headings: "Reseñas de clientes", "Sin reseñas todavía"
- Form: "Escribe tu reseña", "Enviar reseña", textarea placeholder, character counter
- Access states: "Inicia sesión para dejar una reseña", "Solo compradores verificados pueden dejar reseñas", "Ya has dejado una reseña"
- Feedback: "Tu reseña ha sido enviada y será visible tras su aprobación"
- Badge: "Compra verificada"
- Summary: "{count} reseñas" (with singular/plural handling)
- Accessibility: "{rating} de 5 estrellas" (aria-label for star display)

**Namespace `admin.reviews`** (admin panel):
- Table headers: Producto, Usuario, Rating, Comentario, Fecha, Estado
- Filters: Pendientes, Aprobadas, Rechazadas, Todas
- Actions: Aprobar, Rechazar
- Confirm dialogs: "¿Aprobar esta reseña?", "¿Rechazar esta reseña?"

### Review content

Review text is **not translated** — it displays in the language the customer wrote it in.
